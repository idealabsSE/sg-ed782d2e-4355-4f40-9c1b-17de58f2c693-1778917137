## 1. Product Vision

Build a cross-border trust and compliance platform for rentals in Spain, serving Swedish and Spanish users in any combination. The platform answers three questions:

1. **Is this property legitimate?** Regional tourist license, registry presence, compliance status
2. **Is this person credible?** Identity, solvency, documentation, verified profile
3. **Are they actually connected to this property?** Ownership, authority, tenant or guarantor role verified against evidence

> Verify the property. Verify the person. Verify their connection to it.

The product is a unified verification platform with country-specific modules — not a Swedish tool bolted onto a Spanish property lookup.

## 4. Product Layers

The product is organized into three trust layers. All three must be present for a complete verification.

### Layer A — Property Trust

Verify the Spanish property through official sources:
- Regional tourist-rental license (starts with Comunidad Valenciana)
- Open-data datasets for property registration
- National rental registration (NRA/NRUA) — manual/partner-fed in MVP
- Future: direct automated NRA status

### Layer B — Person Trust

Verify people involved in the rental:
- **Swedish identity** via TIC/Authway (MVP) → Signicat (post-MVP); financial evidence via documents; housing/rental profile
- **Spanish identity** via Verifik (MVP) → Signicat (post-MVP); DNI/NIE/passport evidence, solvency documents, ownership/authority evidence
- Both countries: document-backed workflows, derived trust profile summary

### Layer C — Relationship Trust

Verify that the person is actually connected to the property in the claimed role:
- Registered owner via registry evidence (Nota Simple or equivalent)
- Authorized representative (property manager, power of attorney, or mandate)
- Tenant linked to a specific rental case
- Guarantor linked to a specific case with solvency evidence

**This is not optional — it is a first-class product feature.**

## 6. Confirmed Technical Constraints

These are confirmed facts, not assumptions. Code and architecture must reflect them.

### 6.1 GVA open data is the primary property ingestion source (MVP)

Comunitat Valenciana (GVA) publishes open-data CSV datasets for tourist-use dwellings. These contain: signatura, province, municipality, postal code, address, registration date, cadastral reference, capacity fields. This is the MVP ingestion source. Ingestion must be batch/scheduled, not real-time API polling.

### 6.2 GVA SForms is secondary only

SForms-based search (XML-parameter URL encoding, HTML response) works for spot-checking individual properties. Do not use SForms as the primary ingestion strategy. Use it for secondary validation, manual lookup, fallback spot-check.

### 6.3 No single national Spain-wide property API exists

There is no single bulk public API for all Spanish tourist-rental licenses. Use a federated regional ingestion model — start with GVA, add regions as connectors.

### 6.4 NRA/NRUA automation is not confirmed for MVP

There is a national short-term rental registration layer. The public UI is a SPA with Friendly Captcha. A backend domain was identified but no clean public REST endpoint confirmed. MVP treatment: manual entry or partner-fed. Design the data model now — automate later.

### 6.5 Swedish verification — abstraction first

BankID-based identity: use a provider abstraction layer. Do not hardcode a specific BankID SDK. **MVP provider: TIC / Authway.** Credit/financial verification: document-backed workflows first. Direct credit-bureau API integration is deferred from MVP. Build the interface contract — wire the real provider in a separate step.

### 6.6 Spanish person verification is required, not optional

Spanish owners, tenants, and guarantors must all be verifiable. DNI/NIE/passport evidence workflows. Solvency via documents. Property-linked role verification via Nota Simple. **MVP provider: Verifik** for DNI/NIE and CIF/NIF validation.

### 6.7 Nota Simple for ownership/authority verification

To verify whether a claimed owner/host is actually registered, use Nota Simple (Spanish land registry extract). Available via:
- Third-party intermediary (e.g., RegistroPortal, Tinsa, or equivalent) — recommended for MVP
- Direct Registradores access — deferred from MVP

**MVP approach:** manual upload by user OR digital order through an intermediary portal. Direct API integration deferred. The ownership verification module must support the full workflow: order initiation → document receipt → key field extraction → person identity match → result status → reviewer queue for ambiguous cases.

**Ownership verification result states:** `ownership_confirmed` | `mismatch` | `manual_review_needed`

### 6.8 Identity provider strategy — phased

MVP uses two narrow, Spain/Sweden-specific providers:
- **TIC/Authway** for Swedish BankID/Freja eID — simple abstraction, fast integration
- **Verifik** for Spanish DNI/NIE and CIF/NIF — direct Spain-specific endpoints

Post-MVP migration to **Signicat** (consolidated European identity platform) is planned. This migration requires a provider abstraction layer from day one. Signicat's capabilities include: BankID abstraction, biometrics, liveness, data verification, eID orchestration across European markets, and Spanish eID via acquired Electronic IDentification.

**Architectural requirement:** never call TIC, Authway, or Verifik directly from application logic. All calls go through the internal provider interface.

### 6.9 Nota Simple intermediary vs. Verifik/Signicat

Verifik and Signicat handle **person identity** (who is this person?). Nota Simple handles **property ownership** (is this person registered as owner of this property?). These are separate concerns and require separate modules. No identity provider replaces the Nota Simple workflow.

## 7. Multilingual Requirement (ARCHITECTURE CONSTRAINT)

| Language | Status |
|----------|--------|
| English | Default — required from first build |
| Swedish | Required before MVP completion |
| Spanish | Required before MVP completion |

Multilingual support must be built into the app from the first commit, not added later.

### Rules — no exceptions

- All user-facing strings must come from translation resources. No hardcoded UI copy anywhere.
- Email templates, notifications, exported documents, and trust summaries must be locale-aware.
- Shareable trust profiles must be renderable in all three languages.
- Backend error messages and Edge Function responses must be translatable.
- Admin and reviewer screens must support all three languages.
- Language selection: user preference → organization default → browser locale → English fallback

### Engineering implementation rules

- Use a proper i18n framework from the first frontend build (e.g., next-intl, i18next, or equivalent).
- Translation key files must be committed alongside the first component that uses them.
- Database-generated text and dynamic content must use locale-aware templates.
- No MVP feature is done unless it works in English, Swedish, and Spanish.

## 8. Architecture Principles

### Core principles

| Principle | What it means |
|-----------|--------------|
| Supabase-native | Postgres, Auth, RLS, Storage, Edge Functions. Minimize third-party infrastructure. |
| Modular monolith | Single deployable unit. Strong module boundaries. Not premature microservices. |
| Open/closed | Add features by creating modules, not editing existing ones. |
| RLS as primary authorization | Every non-public table has RLS. No client-side auth logic as sole control. |
| Privacy by default | Minimum data collected. Strict separation of public property data vs. private person data. |
| Security by design | Least privilege, default deny, auditability, defense in depth from day one. |
| Multilingual from day one | i18n architecture in place before any user-facing string is hardcoded. |
| Plugin-ready | Connectors, providers, validators, exporters are pluggable modules, not inline logic. |
| Provider-agnostic identity | Identity providers (TIC, Authway, Verifik, Signicat) behind an internal abstraction layer. |
| MCP-first where useful | Use MCP integrations for property lookup, license lookup, and internal AI tooling. |
| Explicit contracts | Modules communicate through defined interfaces, not internal imports. |

### What modular monolith means in practice

- One Supabase project, one database, one deployment
- Feature code is organized into bounded modules in the codebase
- Modules own schema, migrations, RLS policies, Edge Functions, tests, translation keys, public interface
- Adding a new connector or provider: create module → register → add config, migrations, tests → done
- No other module files should need editing when a new module is added

## 9. Supabase Backend Rules

| Rule | Detail |
|------|--------|
| RLS on all tables | Every non-public application table has RLS enabled. No exceptions. |
| `servicerole` never in client | `servicerole` key only in Edge Functions and server-side jobs. Never exposed to browser. |
| Edge Functions for trusted logic | Auth checks, external integrations, sensitive operations, scheduled jobs — all in Edge Functions. |
| Private storage buckets | All user documents in private buckets. Access via signed URLs with expiry. |
| Audit logging | All sensitive data access events written to `accessauditlog`. |
| Secrets in Vault | All API keys, tokens, provider credentials in Supabase Vault or environment secrets. Never hardcoded. |
| Auth + RLS access control | No feature may rely on frontend-only permission checks. |

These rules must be followed in every PR, every Edge Function, every migration.

## 11. Feature Areas

### A. Property Verification

- Resolve property by address, cadastral reference, CRU, license number, or municipality
- Fetch and display regional license data from ingested source
- Return compliance profile with source name, ingestion timestamp, and status
- Support secondary spot-check via SForms lookup
- Display whether NRA registration is present (if data available)

### B. Bulk Ingestion and Source Monitoring

- Scheduled Edge Function to fetch GVA open-data CSV
- Normalize and diff against previous snapshot
- Write new/updated/removed records to `regionallicense`
- Log run results and health status in `connectormetadata`
- Alert ops on anomalies (empty result, schema change, zero rows)
- Connector interface is pluggable — new regions implemented without changing core

### C. Person Verification

**Swedish (TIC/Authway MVP → Signicat post-MVP):**
- Identity via BankID/Freja eID provider abstraction — provider-agnostic interface
- Financial evidence via document upload (payslips, employment certificate, income statements)
- Optional rental history and references
- Derived trust profile summary (shareable, no raw data exposed)

**Spanish (Verifik MVP → Signicat post-MVP):**
- Identity via DNI/NIE/passport evidence — Verifik API for validation in MVP
- If owner/host: Nota Simple or equivalent ownership document (see Feature Area D)
- If tenant: payslips, work contract, bank statements, tax return
- If guarantor: avalista documents, income evidence
- Derived trust profile summary

### D. Ownership and Authority Verification (Nota Simple Module)

- Initiate ownership check for a case (user upload or intermediary order)
- Nota Simple PDF stored in private Storage; metadata in `ownershipverification`
- Key fields extracted: owner name, NIE/DNI, IDUFIR/CRU, property identifiers
- Match against claimed person identity
- Result states: `ownership_confirmed` | `mismatch` | `manual_review_needed`
- Reviewer queue handles ambiguous cases
- Authority verification for property managers (power of attorney, management mandate)

### E. Relationship Trust and Verification Cases

- Link person to property via `verificationcaseparty` + `propertyrelationship`
- Ownership verification against registry evidence
- Authority verification for property managers
- Tenant and guarantor linkage to a specific case
- Role-based trust signal per party

### F. Verification Case Workflow

- Create case with linked property
- Invite parties (owner, tenant, guarantor, co-tenant, representative)
- Each party completes their person verification in their own language
- Evidence aggregated per case
- Internal reviewer can review and decide
- Trust/risk summary generated per case
- Case result shared with parties per RLS-controlled permissions

### G. Admin and Compliance Operations

- Manual review queue for cases requiring human decision
- DSAR intake, tracking, and resolution (access, erasure, portability)
- Retention/deletion schedules and sweep execution
- Vendor registry maintenance (TIC, Authway, Verifik, Signicat, Nota Simple intermediary, etc.)
- Audit log review
- Security incident recording and tracking
- Manual NRA status update (MVP fallback)

### H. Multilingual Experience

- i18n framework in place from first build
- English default, Swedish and Spanish before MVP completion
- All UI, notifications, exports, summaries, and admin screens localized
- Per-user and per-org language preference
- Translation key management as ongoing product operation

### I. Extensibility Layer (Phase 5 and beyond)

- Migrate identity providers from TIC/Authway + Verifik to Signicat
- New regional connectors added as modules
- New identity/financial providers added as modules
- Direct Nota Simple API integration when available
- NRA automated integration path when official API available
- MCP adapter layer for property lookup, license lookup, case tools, AI-assisted review
- Partner API for white-label access

## 12. Key User Flows

### Flow 1 — Property lookup

Input: address / cadastral ref / CRU / license number → resolve property identifier → query `regionallicense` (ingested GVA data or secondary SForms) → return compliance profile, source, timestamp → optional trigger Nota Simple ownership lookup

### Flow 2 — Swedish person verification (TIC/Authway → Signicat)

User initiates → identity step (TIC/Authway BankID/Freja eID abstraction) → financial step: document upload (payslips, employment cert, income proof) → documents stored in private Storage, metadata in `persondocument` → automated rules check → optional manual review → trust profile summary generated (derived, no raw data exposed) → profile shareable via signed link or case attachment

### Flow 3 — Spanish person verification (Verifik → Signicat)

User initiates → identity evidence upload (DNI/NIE/passport) → Verifik API validates document data → role-specific step: owner (Nota Simple upload or intermediary order) / tenant (payslips, work contract, bank statements) / guarantor (avalista docs and income evidence) → documents stored securely, metadata in `persondocument` → reviewer review → trust profile generated

### Flow 4 — Cross-border verification case

Agency creates case → links property → invites owner (ES) + tenant (SE) + optional guarantor → each party completes verification in their own language → relationship layer: owner verified against property registry → reviewer reviews case holistically → trust/risk summary generated → case result shared with parties (RLS-controlled) → all notifications and summaries in correct language per party

### Flow 5 — Regional ingestion (ops)

Scheduled Edge Function triggers → GVA connector → fetch latest open-data CSV → normalize to staging table → diff against last `sourcesnapshot` → write new/updated/removed to `regionallicense` → log run in `connectormetadata` → alert if anomaly

### Flow 6 — Nota Simple ownership verification

User/agency initiates → Nota Simple ordered via intermediary portal or uploaded → PDF stored in private Storage → key fields extracted (owner name, NIE/DNI, IDUFIR) → matched against claimed person identity → result status set (`ownership_confirmed` / `mismatch` / `manual_review_needed`) → reviewer queue if ambiguous → `ownershipverification` record updated → case `trustrisksummary` updated

## 13. Security Requirements

### Principles (non-negotiable)

- **Least privilege** — every role, function, and service gets minimum access needed
- **Default deny** — access must be explicitly granted, not inherited
- **Defense in depth** — multiple independent layers (RLS, Edge Function auth, application)
- **Secure defaults** — restrictive config out of the box; permissive config requires explicit opt-in
- **Auditability** — sensitive operations logged, traceable, reviewable
- **Minimized blast radius** — if a credential or module is compromised, damage is bounded

### Implementation rules

- All private documents: access via signed URLs with short expiry
- Edge Functions: validate auth before any privileged action
- `servicerole` key never leaves server context
- All secrets in Supabase Vault or deployment environment — never hardcoded, never logged
- No highly sensitive data in server logs
- File uploads: type validation, size limits, virus scanning path (non-MVP)
- Ingestion: validate and sanitize all external data before writing to DB
- Admin tooling: separate role, separate RLS policy, full audit trail

## 14. GDPR Requirements

| Principle | Implementation |
|-----------|---------------|
| Lawfulness & transparency | Legal basis documented per processing activity in `dataprocessingrecord` |
| Purpose limitation | Only collect data needed for the specific rental/verification case |
| Data minimization | Prefer derived summaries over raw data storage where possible |
| Storage limitation | Retention/deletion schedules per data category — automated sweep jobs |
| Integrity & confidentiality | Encryption at rest (Supabase default), RLS, private buckets, signed URLs |
| Accountability | Audit log, vendor registry, DSAR workflow, periodic security reviews |

GDPR compliance is mandatory and must shape every schema decision, every feature, every integration.

### Key design rules

- Separate public property data from private person data (different tables, different RLS)
- Do not store raw identity documents longer than necessary
- Shareable trust profiles expose only derived signals, not raw evidence
- Swedish credit reports only where there is a documented, legitimate rental-related need
- DSAR requests must be processable within legal deadlines
- All third-party processors (TIC, Authway, Verifik, Signicat, Nota Simple intermediary) must be registered in `vendorregistry`
