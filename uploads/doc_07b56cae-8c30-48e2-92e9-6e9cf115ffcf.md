# Trust Platform — Development Plan v0.4

## How to Use This File
Provide this file alongside `prd.md` to your AI coding assistant.
Build one phase at a time. Do not start the next phase until the current one passes its checkpoint.
Each step has a clear, testable output.

## Stack
Supabase (Postgres, Auth, RLS, Storage, Edge Functions) · Next.js App Router · TypeScript · Tailwind CSS · next-intl

## Phase 1 — Foundation

**Goal:** Supabase project, auth, org model, i18n, core schema, GDPR baseline, security baseline, module structure.

### Step 1.1 — Supabase project setup
- Initialize Supabase project (Postgres, Auth, Storage, Edge Functions)
- Store env vars in Vault: Supabase URL, anon key, service role key — never hardcode
- Create `.env.local` template with placeholder keys
- **Output:** Supabase project live, env config in place

### Step 1.2 — Next.js scaffold
- Next.js App Router + TypeScript + Tailwind CSS
- Install and configure `next-intl`
- Create locale files: `src/i18n/locales/en.json`, `sv.json`, `es.json`
- Locale detection: user preference → org default → browser → English fallback
- **Output:** App boots, i18n renders "Hello" in EN/SV/ES on a test page

### Step 1.3 — Module folder structure
Create folders under `src/`:
`sharedkernel` `auth` `organizations` `i18n` `properties` `regionallicenses` `nationallicenses` `swedishpersonverification` `spanishpersonverification` `documents` `verificationcases` `connectors` `ownershipverification` `scoringandrules` `privacyandgdpr` `securityandaudit` `admin` `notifications`

Each module gets a `README.md` with module purpose and public interface.
- **Output:** Folder structure committed, no empty files

### Step 1.4 — Core database schema
Migrations for:
- `profiles` (1:1 with `auth.users`)
- `organizations` (org record, locale preference)
- `organization_members` (user ↔ org)
- `organization_roles`
- `properties`
- `access_audit_log`
- `data_processing_record`
- `data_subject_request`
- `vendor_registry`
- **Output:** All migrations apply cleanly, tables visible in Supabase dashboard

### Step 1.5 — RLS baseline
- Enable RLS on every table from Step 1.4
- Default policy: deny-all
- Exception: `properties` — public read for compliance status only
- Users: read own `profiles` row
- Org members: read own org data
- **Output:** RLS tested — unauthenticated user cannot read `profiles` or `organizations`

### Step 1.6 — Auth flow
- Sign-up, sign-in, sign-out via Supabase Auth
- On sign-up: create `profiles` row via Edge Function trigger
- Protect all non-public routes with middleware auth check
- **Output:** Register, login, logout work. Protected routes redirect unauthenticated users.

### Step 1.7 — Connector interface
Define in `src/connectors/interface/`:
```typescript
interface DataConnector {
  fetchSnapshot(): Promise<RawSnapshot>
  normalizeRows(raw: RawSnapshot): NormalizedRow[]
  resolvePropertyIdentifier(row: NormalizedRow): PropertyIdentifier | null
}
```
- **Output:** Interface file committed, documented in module README

### Step 1.8 — Identity provider abstraction interface
Define in `src/sharedkernel/providers/`:
```typescript
interface IdentityProvider {
  startVerification(params: VerificationParams): Promise<string> // verificationId
  getVerificationStatus(verificationId: string): Promise<VerificationStatus>
  normalizeResult(rawResult: unknown): IdentityVerificationRecord
  storeEvidenceReference(personId: string, verificationId: string, provider: string): Promise<void>
}
```
- **Output:** Interface file committed. No provider wired yet.

## Phase 2 — Property Trust Layer

**Goal:** GVA CSV ingestion live, property lookup working, compliance profile returned.

### Step 2.1 — GVA connector
Create `src/connectors/gvacsv/`:
- `fetchSnapshot()` — fetch GVA open-data CSV
- `normalizeRows()` — map CSV to `NormalizedRow`: signatura, province, municipality, postal code, address, registration date, cadastral reference, capacity
- `resolvePropertyIdentifier()` — match normalized row to property record
- **Output:** Unit test: fetch CSV → normalized rows with correct fields

### Step 2.2 — Property data schema
Migrations for:
- `regional_license` (ingested GVA records, links to `properties`)
- `national_license` (NRA/NRUA, nullable, manual MVP)
- `source_snapshot` (raw ingestion record per connector run)
- `connector_metadata` (connector registry, run history, health)
RLS: `regional_license` readable by authenticated users; write only via Edge Function with service role.
- **Output:** Tables created, RLS applied

### Step 2.3 — Ingestion Edge Function (`ingest-gva-csv`)
1. Trigger GVA connector → fetch CSV
2. Normalize rows
3. Diff against last `source_snapshot`
4. Write new/updated/removed records to `regional_license`
5. Log run in `connector_metadata` (timestamp, rows processed, status)
6. Alert ops if: zero rows returned, schema mismatch, or fetch error
- **Output:** Function runs manually, rows ingested, connector_metadata updated

### Step 2.4 — Property lookup API (`lookup-property`)
- Input: address, cadastral reference, CRU, or license number
- Resolve identifier via `resolvePropertyIdentifier`
- Query `regional_license` for matching records
- Return: source, ingestion timestamp, license status, license number
- Handle not-found gracefully
- **Output:** API returns compliance profile for a known GVA property

### Step 2.5 — Property lookup UI
- Property search page (EN/SV/ES via next-intl)
- Input: address or license number
- Display: license status badge, source, timestamp
- Error states: not found, no data available
- **Output:** User can search a property and see compliance status

### Step 2.6 — SForms secondary lookup (optional)
- Manual secondary check only — not primary ingestion
- Use for spot-check or fallback when GVA CSV data is stale
- **Output:** Admin/ops can trigger SForms lookup for a single property

### ✅ Phase 2 Checkpoint
- [ ] GVA connector fetches and normalizes CSV correctly
- [ ] Ingestion Edge Function runs, diffs, writes to regional_license
- [ ] connector_metadata logs every run
- [ ] Ops alert fires on anomaly
- [ ] Property lookup API returns compliance profile
- [ ] Property search UI works in EN, SV, ES

## Phase 3 — Person Trust Layer

**Goal:** Swedish + Spanish person verification live; Nota Simple ownership workflow; document storage; reviewer queue.

### Step 3.1 — Person data schema
Migrations for:
- `person`
- `identity_verification` (1:many — per method/provider/country)
- `financial_profile` (1:1 derived summary)
- `person_document` (1:many — metadata only, file in Storage)
- `person_trust_profile` (1:1 derived shareable summary)
RLS: person sees own records; agency sees org-scoped; reviewer sees assigned cases.
- **Output:** Tables created, RLS applied and tested

### Step 3.2 — Document storage
- Private Supabase Storage bucket: `person-documents`
- Upload Edge Function: validate file type (PDF/image), enforce size limit, store, write metadata to `person_document`
- Signed URL Edge Function: short-expiry URL, log access to `access_audit_log`
- **Output:** Upload works, signed URL returned, access logged

### Step 3.3 — TIC/Authway provider (Swedish identity MVP)
Create `src/swedishpersonverification/providers/tic/`:
- Implement `IdentityProvider` interface
- `startVerification()` — initiate BankID/Freja eID via TIC/Authway API
- `getVerificationStatus()` — poll or webhook for result
- `normalizeResult()` — map to `IdentityVerificationRecord`
- Store evidence reference only (no raw data in DB)
- **Output:** Tested with TIC/Authway sandbox or mock

### Step 3.4 — Swedish person verification flow
- Flow: identity step (TIC/Authway) → document upload (payslips, employment cert, income proof) → automated rules check → optional manual review queue
- EN and SV
- On completion: generate `person_trust_profile` (derived signals only)
- Profile shareable via signed link or case attachment
- **Output:** Swedish user completes full verification flow end-to-end

### Step 3.5 — Verifik provider (Spanish identity MVP)
Create `src/spanishpersonverification/providers/verifik/`:
- Implement `IdentityProvider` interface
- `startVerification()` — submit DNI/NIE/passport to Verifik
- `getVerificationStatus()` — get validation result
- `normalizeResult()` — map to `IdentityVerificationRecord`
- **Output:** Verifik adapter tested with sandbox or mock

### Step 3.6 — Spanish person verification flow
Role-specific flows:
- **Owner/host:** identity (Verifik) + Nota Simple ownership (Step 3.7)
- **Tenant:** identity (Verifik) + payslips, work contract, bank statements
- **Guarantor:** identity (Verifik) + avalista docs + income evidence
- EN and ES
- On completion: generate `person_trust_profile`
- **Output:** Spanish user (owner, tenant, guarantor) completes their flow

### Step 3.7 — Nota Simple ownership workflow
Create `src/ownershipverification/`:
- User uploads or orders via intermediary
- PDF stored in private Storage, metadata in `ownership_verification` table
- Extract: owner name, NIE/DNI, IDUFIR/CRU, property identifiers
- Match extracted fields against claimed person identity
- Set result: `ownership_confirmed` | `mismatch` | `manual_review_needed`
- Route `manual_review_needed` to reviewer queue
- Update `ownership_verification` + trigger `trust_risk_summary` update
- **Output:** Ownership flow works end-to-end, reviewer queue shows pending items

### Step 3.8 — Reviewer queue
- Internal reviewer UI (admin, EN sufficient for MVP)
- Show pending cases requiring manual review
- Reviewer actions: view evidence, make decision, write notes, set outcome
- Decision stored in `review_outcome`, triggers case trust summary update
- **Output:** Reviewer sees queue, opens case, makes decision, closes case

- [ ] Swedish verification flow complete end-to-end
- [ ] Verifik abstraction works (sandbox or mock)

## Phase 4 — Relationship and Case Workflows

**Goal:** Full multi-party verification case; per-party notifications; complete review flow; trust/risk summary.

### Step 4.1 — Case schema
Migrations for:
- `verification_case` (linked to property)
- `verification_case_party` (person + role per case)
- `property_relationship` (person ↔ property ↔ role ↔ evidence)
- `authority_verification`
- `trust_risk_summary`
RLS: parties see own case data; agency sees org-scoped; reviewer sees assigned cases.
- **Output:** Tables created, RLS tested

### Step 4.2 — Case creation
- Agency or owner creates case
- Links property (from property lookup)
- Sets roles: owner, tenant, optional co-owners, optional guarantor
- System generates invitation per party
- **Output:** Case created, parties listed with pending status

### Step 4.3 — Party invitation and onboarding
- Email invitation per party (locale-aware)
- Party follows link → completes their person verification flow (Phase 3)
- Status updates: invited → in-progress → complete
- **Output:** Invitation sent in correct language; party completes verification via link

### Step 4.4 — Ownership and authority on case
- Owner role: trigger Nota Simple ownership check vs linked property
- Manager/rep role: upload power of attorney or management mandate, extract and match
- Result updates case party status
- **Output:** Case shows ownership confirmed/unconfirmed per owner party

### Step 4.5 — Trust/risk summary
- Aggregate evidence across all parties + property
- Generate `trust_risk_summary`: overall signal, per-party signals, open issues
- Update on each status change or reviewer decision
- **Output:** Case detail shows trust/risk summary, updates when evidence changes

### Step 4.6 — Reviewer holistic review
- Full case view: all parties, all evidence, all sub-summaries
- Reviewer makes holistic decision, writes notes, sets `review_outcome`
- Final `trust_risk_summary` updated on close
- **Output:** Reviewer closes a full cross-border case end-to-end

### Step 4.7 — Case result sharing
- RLS-controlled: owner sees tenant profile summary, not raw data
- Export case summary as PDF — locale-aware (EN/SV/ES per requesting party)
- Signed URL for shared case result with expiry
- **Output:** Each party sees exactly what their role permits

### Step 4.8 — Notifications
- Email notifications minimum for MVP
- All templates locale-aware (EN/SV/ES)
- Triggers: invitation sent, verification complete, case updated, review decision made
- **Output:** All notification templates work in all three languages

### ✅ Phase 4 Checkpoint
- [ ] Case creation and party invitation work end-to-end
- [ ] Each party type verifies via invitation flow
- [ ] Ownership verification linked to case property
- [ ] trust_risk_summary generated and updates correctly
- [ ] Reviewer completes full case review
- [ ] Case result shared per RLS permissions
- [ ] All notifications work in EN, SV, ES
- [ ] Full cross-border case (ES owner + SE tenant + guarantor) tested end-to-end
