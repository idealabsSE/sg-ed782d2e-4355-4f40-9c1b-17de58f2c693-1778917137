# Trust Platform — PRD v0.5

## What This App Does
Cross-border rental verification SaaS for Spain. Three questions:
1. Is this property legitimate? (regional tourist license, registry compliance)
2. Is this person credible? (identity, solvency, document-backed evidence)
3. Are they actually connected to this property? (ownership, authority, or tenant role)

## Stack
- **Backend:** Supabase (Postgres, Auth, RLS, Storage, Edge Functions)
- **Frontend:** Next.js App Router + TypeScript + Tailwind CSS
- **i18n:** next-intl (EN default, SV + ES required before MVP sign-off)
- **Architecture:** Modular monolith. Add features by creating modules, not editing existing ones.

## Product Layers (All Required for MVP)

### Layer A — Property Trust
- GVA open-data CSV ingestion (Comunitat Valenciana regional tourist licenses)
- CSV fields: signatura, province, municipality, postal code, address, registration date, cadastral reference, capacity
- SForms is secondary only (manual spot-check, not primary ingestion)
- NRA/NRUA national registration: manual entry for MVP, automate post-MVP

### Layer B — Person Trust
- **Swedish identity:** TIC/Authway behind `IdentityProvider` abstraction (BankID/Freja eID) + financial docs (payslips, employment cert)
- **Spanish identity:** Verifik (DNI/NIE/passport validation) + role-specific evidence docs
- Provider abstraction is mandatory — never call TIC, Authway, or Verifik directly from app logic
- Post-MVP: both replaced by Signicat via config switch, no core code changes

### Layer C — Relationship Trust
- Nota Simple PDF (ownership): upload or intermediary order. Extract owner name, NIE/DNI, IDUFIR/CRU. Match vs claimed identity. States: `ownership_confirmed` | `mismatch` | `manual_review_needed`
- Authority verification: power of attorney / management mandate for manager/rep role
- Tenant or guarantor linked to case with evidence

## Data Model

```
-- Property
property → regional_license (1:many) → national_license (1:1, nullable)
         → source_snapshot (audit) → connector_metadata (connector health)

-- Person
person → identity_verification (1:many) → financial_profile (1:1)
       → person_document (1:many, metadata only) → person_trust_profile (1:1)

-- Case
verification_case → verification_case_party → property_relationship
                 → ownership_verification → authority_verification
                 → review_outcome → trust_risk_summary

-- Org + Auth
auth.users ↔ profiles → organization_members → organizations → organization_roles

-- Compliance
data_processing_record · data_subject_request · vendor_registry
access_audit_log · security_incident · security_review_record
```

## Key User Flows

**Flow 1 — Property Lookup**
address/cadastral/CRU → resolve identifier → query regional_license → compliance profile (source, timestamp, status)

**Flow 2 — Swedish Person Verification**
Start → BankID/Freja eID via TIC/Authway abstraction → document upload → private Storage + person_document metadata → rules check → optional manual review → person_trust_profile → shareable signed link

**Flow 3 — Spanish Person Verification**
Start → DNI/NIE/passport → Verifik validation → role branch:
- Owner: Nota Simple ownership check
- Tenant: payslips + work contract + bank statements
- Guarantor: avalista docs + income evidence
→ private Storage → reviewer review → person_trust_profile

**Flow 4 — Cross-Border Case**
Agency creates case → links property → invites parties (ES owner + SE tenant + optional guarantor) → each verifies in their language → ownership verified → reviewer reviews → trust_risk_summary → shared per RLS

**Flow 5 — GVA Ingestion**
Scheduled Edge Function → GVA connector → fetch CSV → normalize → diff vs source_snapshot → write to regional_license → log to connector_metadata → alert ops on anomaly

**Flow 6 — Nota Simple Ownership**
Upload/order PDF → private Storage → extract fields → match vs claimed identity → set result state → reviewer queue if ambiguous → update ownership_verification + trust_risk_summary

## Non-Negotiable Rules

### Supabase
- RLS on every non-public table — no exceptions
- `service_role` key only in Edge Functions — never in browser
- All user documents in private Storage — signed URLs with expiry
- All sensitive access logged to `access_audit_log`
- All secrets in Supabase Vault — never hardcoded, never logged

### i18n
- No hardcoded UI strings anywhere
- Email templates, notifications, exports, trust summaries must be locale-aware
- Locale resolution: user preference → org default → browser → English fallback
- No feature is done until it works in EN, SV, and ES

### Architecture
- Provider abstraction is mandatory for all identity providers
- Modules communicate through defined interfaces, not internal imports
- New tables default to RLS deny-all
- No client-side auth as sole access control

## GDPR
| Principle | Implementation |
|-----------|---------------|
| Lawfulness | Legal basis per activity in `data_processing_record` |
| Data minimization | Store derived signals, not raw copies where possible |
| Storage limitation | Automated retention/deletion Edge Functions per data category |
| DSAR | `data_subject_request` table + Edge Function handler |
| Vendor registry | All third-party processors in `vendor_registry` |
